# 协议文档生成

根据协议名称生成详细的协议设计文档。

## 执行流程

### 第一步：确认协议名称和文档路径

如果用户只提供协议名称（如 tcp、udp、icmp 等），则：
- 协议名称统一转为小写
- 文档路径默认为：`docs/protocols/{protocol}.md`

如果用户未提供协议名称，使用 AskUserQuestion 询问。

### 第二步：搜索协议标准文档

使用 WebSearch 工具搜索该协议的官方 RFC 文档和相关信息，关键词格式：`{protocol} protocol RFC specification`。

根据搜索结果，获取：
- RFC 编号（如 RFC 791 for IPv4）
- 协议全称
- 发布年份和背景
- 核心功能描述

### 第三步：生成协议文档

按照以下章节结构生成文档：

```markdown
# {协议全称}详细设计文档

## 1. 协议概述

### 1.1 背景与历史

**协议全称和定义：**
- 协议的全称
- 在 TCP/IP 协议栈中的层级位置
- 核心功能概述

**为什么需要该协议？**

说明该协议解决的问题和存在的必要性。

**历史背景：**
- RFC 编号和发布时间
- 协议的发展演进
- 相关的补充 RFC

### 1.2 设计原理

该协议的核心设计思想和基本原理。

使用 ASCII 图或流程图展示协议的工作机制。

**关键特点：**
- 列举 3-5 个核心特点
- 每个特点用一句话说明

## 协议 RFC 参考映射

根据协议特性，可以对章节进行适当调整：

| 协议类型 | 必选章节 | 可选/调整章节 |
|---------|---------|--------------|
| 数据链路层 | 1, 2, 4, 5, 6, 8, 9, 10 | 3(状态机), 7(安全) |
| 网络层 | 1, 2, 4, 5, 6, 8, 9, 10 | 3(状态机), 7(安全) |
| 传输层 | 1, 2, 3, 4, 5, 6, 8, 9, 10 | 7(安全) |
| 应用层协议 | 1, 2, 4, 5, 8, 10 | 3(状态机), 6, 7, 9 |

**注：** TCP 协议必须包含完整的状态机设计章节。

## 项目特定约束

CoreNet 项目的协议文档应遵循以下约定：
1. 使用中文编写
2. 遵循零外部依赖原则（仅使用 Rust 标准库）
3. 采用纯模拟模型（无真实网络接口）
4. 优先考虑可读性和学习价值
5. 遵循统一的文档风格
