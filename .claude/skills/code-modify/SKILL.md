# 代码修改

根据用户需求对现有代码进行修改，包括修复 bug、优化改进、功能调整等。

## 执行流程

### 第一步：收集修改需求

根据用户的修改描述，判断信息是否充分。如果信息不足，通过追问明确以下方面：

**必需信息**：
- 要修改的模块/文件路径
- 修改的具体内容或目标
- 当前问题或需要改进的地方

**可选追问**（视情况）：
- 修改的触发场景是什么？（什么情况下发现的问题）
- 期望的修改效果是什么？
- 是否涉及接口变更？是否会影响其他模块？
- 是否需要保持向后兼容？
- 是否有性能或安全方面的考虑？

**使用 AskUserQuestion 的场景**：
- 修改目标不明确时
- 存在多种修改方案时
- 涉及接口变更需要确认影响范围时
- 需要用户在多个选项中做选择时

### 第二步：定位修改位置

根据用户描述，定位需要修改的代码位置：

**如果用户指定了具体文件**：
1. 使用 Read 读取该文件
2. 分析文件结构，找到相关代码段

**如果用户只描述了模块或功能**：
1. 使用 Glob 在 `src/` 目录搜索相关模块文件
2. 使用 Grep 搜索关键函数名、结构体名或相关关键词
3. 读取匹配的文件，定位具体代码

### 第三步：分析影响范围

分析修改可能影响的其他部分：

**查找依赖此代码的其他模块**：
1. 使用 Grep 搜索被修改函数/结构体的引用
2. 搜索 pub 类型和 pub 函数的使用位置
3. 分析调用链，确定影响范围

**如果涉及其他模块**：
1. 到 `docs/design/` 查看相关模块的设计文档
2. 到 `src/` 查看相关模块的代码实现
3. 理解模块间的接口和交互方式

### 第四步：查看相关设计文档和代码

**查看设计文档**：
- 使用 Glob 列出 `docs/design/*.md` 中的设计文档
- 读取相关模块的设计文档，了解：
  - 模块的职责和定位
  - 数据结构的定义
  - 接口的约定
  - 错误处理策略

**查看相关代码实现**：
- 查看依赖模块的公共接口
- 参考现有代码的实现风格
- 理解模块间的数据传递方式

### 第五步：制定修改计划

使用 TodoWrite 创建任务清单，按以下顺序组织：

1. 理解当前实现和问题所在
2. 设计修改方案
3. 修改代码（如果涉及多个文件，按依赖顺序修改）
4. 验证编译（cargo build）
5. 如有编译错误，修复并重新编译

### 第六步：执行修改

按照以下原则执行修改：

**修改规范**：
- **保持代码风格一致**：参考现有代码的风格
- **遵循 Rust 惯用法**：使用合理的命名、所有权、借用规则
- **添加必要的注释**：如果修改逻辑复杂，添加说明注释
- **错误处理**：使用 `Result<T, E>` 模式，妥善处理错误

**接口变更**：
- 如果修改了 pub 接口，检查所有调用位置
- 如果需要保持兼容性，考虑保留旧接口并标记 deprecated
- 更新相关模块的调用代码

**跨模块修改**：
- 先修改被依赖的模块（下层模块）
- 再修改依赖方（上层模块）
- 确保模块间的接口约定保持一致

### 第七步：编译验证

使用 Bash 运行编译检查：
```bash
cargo build
```

如果编译失败，分析错误信息并修复：
- 类型错误：检查修改后的类型是否匹配
- 缺少导入：添加必要的 use 语句
- 接口不匹配：检查调用方与被调用方的参数和返回值
- 依赖问题：确保模块间的引用关系正确

### 第八步：向用户确认修改结果

完成修改后，向用户报告：
- 修改了哪些文件
- 修改的主要内容
- 编译是否通过
- 是否需要进一步调整

## 严格遵循的原则

1. **明确修改需求**
   - 通过追问确保理解用户的修改意图
   - 明确修改的具体位置和内容
   - 确认修改的影响范围

2. **分析影响范围**
   - 查找被修改代码的所有引用
   - 理解模块间的依赖关系
   - 确保修改不会破坏现有功能

3. **参考设计文档和现有代码**
   - 查阅 `docs/design/` 中的相关设计文档
   - 查看 `src/` 中相关模块的代码实现
   - 保持与项目架构和设计一致

4. **保持代码质量**
   - 遵循 Rust 编码规范
   - 保持代码风格一致
   - 确保编译通过

## 与用户交互的场景

**需要追问的情况**：
- 修改目标不明确
- 存在多种修改方案且需要用户选择
- 涉及接口变更且影响范围不确定
- 修改可能导致兼容性问题

**需要查看设计文档的情况**：
- 修改涉及多个模块
- 需要理解模块间的交互方式
- 需要了解接口的设计意图

**需要查看其他模块代码的情况**：
- 需要理解如何使用其他模块的接口
- 需要参考类似功能的实现方式
- 需要确保修改不影响其他模块

## 完成标准

修改完成时应该满足：
- 按照用户需求完成了代码修改
- 修改后的代码编译通过（cargo build）
- 保持了与现有代码风格一致
- 如果涉及接口变更，相关调用已同步更新
- 向用户报告了修改内容和结果

## 不包含的内容

- 单元测试用例的编写或修改（除非用户明确要求）
- 集成测试代码
- 性能基准测试
- README 文档更新（除非涉及公共接口变更）
