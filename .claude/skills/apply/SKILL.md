# 代码实现

根据设计文档实现 Rust 代码，严格按照设计规范进行编码。

## 执行流程

### 第一步：确认路径信息

使用 AskUserQuestion 询问用户以下信息（如果未指定）：

1. **设计文档路径**：markdown 格式的设计文档文件路径
2. **实现代码存放路径**：源代码目录路径（如 `src/common/` 或 `src/protocols/ethernet/`）

### 第二步：读取并分析设计文档

读取设计文档，提取以下关键信息：

**必需提取的信息**：
- 模块/功能名称
- 核心数据结构定义（结构体字段、类型）
- 接口定义（Trait 和函数签名）
- 模块结构（文件组织方式）
- 错误类型定义

**理解设计意图**：
- 各数据结构的作用和职责
- 接口的功能和参数/返回值含义
- 数据流向和处理逻辑
- 错误处理策略

### 第三步：检查相关模块的设计文档

从设计文档中识别与本模块有交互的其他模块，到 `docs/design/` 目录查看这些模块的设计文档：

**查找相关设计文档**：
- 使用 Glob 在 `docs/design/` 中搜索相关模块的 `.md` 文件
- 重点关注：模块定位、数据流向、接口定义部分

**了解模块间关系**：
- 上层模块如何调用本模块（本模块需要暴露哪些接口）
- 本模块如何调用下层模块（需要使用哪些外部接口）
- 数据结构在各模块间如何传递

**记录依赖信息**：
- 需要引用的其他模块的类型
- 需要调用的其他模块的函数/方法
- 需要遵循的接口约定

### 第四步：检查相关模块的代码实现

到 `src/` 目录查看相关模块的代码实现：

**查看公共接口**：
- 使用 Grep 搜索 `pub fn`、`pub struct`、`pub enum`、`pub trait`
- 了解其他模块暴露的公共 API

**了解数据结构**：
- 查看相关模块定义的公共类型
- 理解如何使用这些类型（构造、访问、修改）

**参考实现风格**：
- 参考现有代码的错误处理方式
- 参考现有代码的注释风格
- 参考现有代码的模块组织方式

### 第五步：检查目标路径现有代码

检查目标路径是否已存在代码：
- 如果存在，读取现有代码，了解已实现的部分
- 对比设计文档，确定需要新增、修改或补充的内容

### 第六步：制定实现计划

使用 TodoWrite 创建任务清单，按以下顺序组织：

1. 创建/更新模块结构（mod.rs 等）
2. 实现核心数据结构
3. 实现错误类型定义
4. 实现接口和核心逻辑
5. 添加必要的 pub use 导出

### 第七步：实现代码

严格按照设计文档实现代码，遵循以下原则：

**实现规范**：
- **严格按照设计文档**：数据结构字段、函数签名、类型必须与设计文档一致
- **与现有代码集成**：
  - 引用相关模块的类型时使用正确的路径
  - 调用其他模块接口时遵循其约定
  - 保持与现有代码风格一致
- **添加文档注释**：为 pub 结构体、pub 函数添加文档注释
- **错误处理**：使用 `Result<T, E>` 模式，按照设计文档的错误处理策略实现
- **遵循 Rust 惯用法**：使用合理的命名、所有权、借用规则

**代码风格**：
- 使用 `cargo fmt` 格式化
- 通过 `cargo clippy` 检查
- 保持代码简洁清晰

**模块组织**：
- 按照设计文档的模块结构创建文件
- 在 mod.rs 中添加模块声明和 pub use 导出
- 确保模块层次清晰

### 第八步：编译验证

使用 Bash 运行编译检查：
```bash
cargo build
```

如果编译失败，分析错误信息并修复：
- 类型不匹配：检查是否严格按照设计文档实现
- 缺少导入：添加必要的 use 语句
- 语法错误：修正代码问题
- 依赖问题：确保只使用标准库（零外部依赖原则）

## 严格遵循的原则

1. **严格按照设计文档实现**
   - 数据结构字段名称、类型必须与设计文档一致
   - 函数签名（参数、返回值）必须与设计文档一致
   - 模块结构和文件组织按照设计文档执行

2. **理解并遵循模块间关系**
   - 查看相关模块的设计文档（`docs/design/`），理解模块间的交互方式
   - 查看相关模块的代码实现（`src/`），了解可用的公共接口
   - 正确引用和使用其他模块的类型和函数
   - 保持与现有代码风格一致

3. **不生成单元测试用例**
   - 不在代码文件中添加 `#[cfg(test)]` 测试模块
   - 不添加 `#[test]` 测试函数
   - 不生成测试辅助函数或测试数据

4. **零外部依赖**
   - 只使用 Rust 标准库（`std`）
   - 不引入外部 crate

5. **代码质量**
   - 添加必要的文档注释
   - 保持代码简洁清晰
   - 通过 `cargo build` 编译

## 与用户交互的场景

**需要询问用户的情况**：
- 设计文档中有模糊或矛盾的内容
- 设计文档缺少必要的实现细节
- 现有代码与设计文档不一致，需要确认如何处理

## 完成标准

实现完成时应该满足：
- 所有设计文档中定义的数据结构已实现
- 所有设计文档中定义的接口已实现（函数体有基本实现）
- 正确引用和使用相关模块的类型和接口
- 代码可以通过 `cargo build` 编译
- 模块导出（pub use）符合设计文档要求
- 与现有代码风格保持一致
- 没有生成任何单元测试代码

## 不包含的内容

- 单元测试用例（`#[cfg(test)]`、`#[test]`）
- 集成测试代码
- 性能基准测试
- 使用示例代码
- README 文档
