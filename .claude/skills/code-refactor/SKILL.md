# 代码重构

对现有 Rust 代码进行重构，优化代码质量和可维护性。

## 重构规则

1. **不改变原有逻辑**：只优化实现方式，保持功能不变
2. **优化函数调用链**：检查函数调用链，将串行的短函数合并成一个
3. **检查封装性**：不需要对外提供的函数改为 private
4. **检测过度封装**：识别仅转发调用、无额外逻辑的包装函数
5. **消除重复代码**：查找相似逻辑的重复实现，提取公共函数
6. **更新函数注释**：简要说明函数功能、入参和返回值即可
7. **清理无用代码**：去除 dead code 和未使用变量

## 执行流程

### 第一步：确认重构范围

使用 AskUserQuestion 询问用户以下信息（如果未指定）：

1. **重构目标**：目录、模块或文件路径（如 `src/common/queue.rs` 或 `src/protocols/`）
2. **重构范围确认**：确认是否包括相关测试文件

### 第二步：读取并分析代码

读取目标代码文件，进行全面分析：

**检查函数调用链**：
- 识别仅被单一位置调用的短函数（<5行）
- 识别仅转发调用、无额外逻辑的包装函数
- 记录可合并的函数组合

**检查过度封装**：
- 识别仅转发参数、无额外逻辑的包装函数（如 `fn wrapper(a, b) { inner(a, b) }`）
- 识别仅改变参数顺序的适配函数
- 识别仅做类型转换的薄封装层
- 检查是否存在多个中间层函数，实际逻辑可以内联

**检查重复代码**：
- 查找相似的代码片段（如重复的解析逻辑、错误处理模式）
- 识别可以用泛型或 trait 统一的重复实现
- 检查是否存在多处相似的验证、转换逻辑
- 查找重复的错误构造模式

**检查封装性**：
- 使用 Grep 搜索所有 `pub fn`、`pub struct`、`pub enum`、`pub trait`
- 在项目范围内搜索这些公共项的使用情况
- 识别未被外部使用的 public 项

**检查代码质量**：
- 识别冗余的结构体/枚举（如仅用于单一薄封装场景）
- 查找未使用的变量、导入、函数
- 查找 dead code（永远不会被执行的代码分支）

**检查注释**：
- 识别过度冗长的注释（如使用大量 "# 参数"、"# 返回" 分隔标记）
- 识别缺失必要注释的公共 API

### 第三步：生成重构建议报告

使用 TodoWrite 创建重构清单，向用户展示以下分析结果：

```
=== 重构分析报告 ===

【可优化的函数调用链】
- 函数名：xxx::yyy()
  当前：5行分隔式注释 + 3行逻辑
  建议：简化为1行注释
  理由：函数功能简单，注释过于冗长

【可合并的函数】
- with_config() -> new()
  当前：with_config() 仅调用 new(config.capacity)
  建议：移除 with_config()，直接使用 new()

【过度封装检测】
- 函数名：module_a::process()
  当前：仅转发调用 module_b::inner_process()，无额外逻辑
  建议：直接调用 inner_process()，移除中间层
  影响：2处调用点

【重复代码检测】
- 位置：module_a.rs:42-58, module_b.rs:71-87
  当前：两处相同的 ARP 报文解析逻辑（16行重复）
  建议：提取公共函数到 common/parse.rs
  影响：可减少约 14 行代码

【可改为 private 的项】
- struct QueueConfig: 字段 capacity 为 pub，但不需要外部构造

【可移除的 dead code】
- test_default_config: 因移除 QueueConfig 而失效的测试

【可简化的注释】
- 函数：enqueue()
  当前：7行分隔式注释
  建议：1行简洁注释
```

### 第四步：等待用户确认

使用 AskUserQuestion 询问用户：
```
是否按上述建议执行重构？
- [是] 执行重构
- [否] 取消重构
- [部分] 选择性执行部分建议
```

### 第五步：执行重构

获得用户确认后，按以下顺序执行重构：

1. **移除冗余代码**：删除 dead code、未使用的变量/导入
2. **合并函数**：将串行的短函数合并
3. **移除过度封装**：删除仅转发调用的包装函数，更新调用点
4. **消除重复代码**：提取公共函数，统一相似逻辑的实现
5. **调整封装性**：将不需要对外提供的函数/字段改为 private
6. **更新注释**：简化冗长的注释，补充缺失的注释
7. **同步修改**：更新 mod.rs、lib.rs 中的相关导出
8. **更新测试**：移除或更新失效的测试用例

### 第六步：验证编译

执行编译和测试验证：

```bash
# 编译检查
cargo check

# 运行测试
cargo test

# 如有需要，运行 clippy 检查
cargo clippy
```

如果编译/测试失败，分析错误信息并修复问题。

### 第七步：输出重构摘要

向用户报告重构结果：

```
=== 重构完成 ===

修改文件：
- src/common/queue.rs: 488 -> 456 行（减少 32 行）
- src/common/mod.rs: 移除 QueueConfig 导出
- src/lib.rs: 移除 QueueConfig 导出

测试结果：
✓ 27 个测试通过
```

## 重构原则

**必须遵循**：
1. 不改变原有功能逻辑
2. 所有测试必须通过
3. 保持代码风格一致性
4. 编译无警告（除 clippy 的合理建议外）

**禁止操作**：
- 修改公共接口签名（除非确认无外部使用）
- 删除可能被外部使用的公共 API
- 改变数据结构的语义
- 引入新的依赖项

## 完成标准

重构完成时应该满足：
- 代码编译通过（`cargo check`）
- 所有测试通过（`cargo test`）
- 代码行数减少或可读性提升
- 注释简洁准确
- 无 dead code 和未使用变量

## 与用户交互的场景

**必须询问用户的情况**：
- 执行重构前，必须展示分析报告并获取确认
- 发现可能影响外部使用的公共 API 修改时
- 重构方向存在多种合理方案时
